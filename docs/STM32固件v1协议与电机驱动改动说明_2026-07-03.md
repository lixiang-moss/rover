# STM32 固件 v1 协议与电机驱动改动说明

> 文档用途：交给 STM32、驱动器和电机负责人做代码审查、CubeIDE 构建及实物验证
> 固件目录：`STM32/`
> 目标芯片：STM32G474RET6 / STM32G474RETx
> 修改日期：2026-07-03

## 1. 结论

本次已经把 STM32 工程从旧版嵌套 JSON 测试程序，改成与当前 Raspberry Pi / ROS 2 一致的 `v=1` 固件实现：

- 接收紧凑 W 帧并校验 CRC32。
- 返回可被 Pi 直接解析的 ACK 和 STATUS。
- 以约 5 Hz 主动发送 STATUS。
- 超过 0.5 秒没有合法 W 帧时停止输出。
- 按四轮固定顺序处理 4 个转向角和 4 个行走速度。
- 按 MKS SERVO57D Modbus RTU 协议生成绝对多圈坐标命令。
- 按 BLD-305S 原始手册生成速度、正反转、停止和故障读取命令。
- 驱动器请求必须收到地址、功能码和 CRC16 均正确的响应，才算成功。
- 对每个轮组执行“先转向到位，再允许行走电机启动”。

但当前提交**默认禁止真实电机动作**。STM32 负责人必须核对第 6 节中的实物参数和安全输入，再修改 `mwrs_config.h` 的确认项。这样做是为了防止尚未确认的零位、方向或驱动器协议在烧录后直接驱动电机。

## 2. 实现依据

### 2.1 Pi 与 STM32 接口

接口以以下项目文档为准：

- `docs/Pi与STM32接口对接说明.md`
- `docs/STM32负责人USB通信与协议修改交接说明.md`

Pi 发送：

```text
<紧凑 JSON>*<8 位大写十六进制 CRC32>\n
```

W 帧固定字段为 `v,t=W,q,m,e,s,w`，四轮顺序固定为：

```text
front_left, front_right, rear_left, rear_right
```

每个轮组固定为：

```text
[enabled, angle_rad, velocity_mps, steering_limit_radps, acceleration_limit_mps2]
```

### 2.2 BLD-305S

依据项目原始手册：

```text
Group 3.1 - Final Submission/Data Sheets/BLD-305S_Manual (1).pdf
```

本次使用的寄存器：

| 地址 | 操作 | 用途 |
|---|---|---|
| `0x0056` | 写，功能码 `0x06` | 目标电机转速，单位 rpm |
| `0x005F` | 读，功能码 `0x03` | 实际电机转速 |
| `0x0066` | 写，功能码 `0x06` | 0 停止、1 正转、2 反转、3 制动停止 |
| `0x0066` | 读，功能码 `0x03` | 当前运行状态 |
| `0x0076` | 读，功能码 `0x03` | 驱动器故障码 |

BLD-305S 总线按 `115200, 8N1` 配置，驱动器地址改为 `1-4`，不再使用旧代码中的 `5-8`。

### 2.3 MKS SERVO57D

依据 Makerbase 官方仓库中的 Modbus RTU V1.0.9 手册：

<https://github.com/makerbase-motor/MKS-SERVO42D-57D/tree/master/User%20Manual/V1.0.9>

本次使用的寄存器：

| 地址 | 操作 | 用途 |
|---|---|---|
| `0x0031` | 读，功能码 `0x04` | 累计多圈坐标，6 字节有符号值 |
| `0x003B` | 读，功能码 `0x04` | homing 状态 |
| `0x003E` | 读，功能码 `0x04` | 堵转状态 |
| `0x00F3` | 写，功能码 `0x06` | 总线模式使能 |
| `0x00F5` | 写，功能码 `0x10` | 绝对坐标运动 |
| `0x00F7` | 写，功能码 `0x06` | 紧急停止 |

注意：以上寄存器对应 MKS Modbus RTU 固件 V1.0.9。实物驱动器的固件版本必须核对；如果版本不同，STM32 负责人必须先确认寄存器兼容性，不能直接带载运行。

## 4. 协议和安全逻辑

### 4.1 接收

- LPUART1 使用中断逐字节接收。
- 中断只把字节写入 1024 字节环形缓冲区。
- 主循环按换行符切帧，不在中断内解析 JSON。
- 完整帧上限为 512 字节。
- 缓冲溢出或超长帧会停止输出，并上报 `1002`。

### 4.2 严格校验

固件会拒绝：

- CRC32 错误或不是 8 位大写十六进制。
- 不是 `v=1`、`t=W` 的帧。
- 多字段、少字段或重复字段。
- q 不是 uint32。
- e、s、轮组 enabled 不是数字 0/1。
- 轮组数量不是 4，或每个轮组不是 5 个值。
- NaN、Inf、负限幅和超过硬边界的目标。
- RAW_WHEEL_TEST 同时使能的轮组数不是 1。
- CRAB 或 SPIN_IN_PLACE 在真实使能时没有使能全部四个轮组。

固定测试向量仍为：

```text
{"e":1,"m":3,"q":42,"s":0,"t":"W","v":1,"w":[[1,0.1,0.08,0.15,0.05],[0,0.0,0.0,0.15,0.05],[0,0.0,0.0,0.15,0.05],[0,0.0,0.0,0.15,0.05]]}*58782C06
```

### 4.3 ACK 与 STATUS

合法且被接受的命令返回：

```text
{"fc":0,"ok":1,"q":42,"t":"A","v":1}*88B875B3
```

正常状态示例：

```text
{"es":0,"fc":0,"on":1,"q":42,"t":"S","to":0,"v":1}*F18BB4D1
```

`on=1` 只有在真实输出已解锁、硬件急停输入已确认、要求的 MKS/BLD 均在线、无驱动故障且 homing 完成时才会出现。

### 4.4 0.5 秒 watchdog

超过 500 ms 没有合法 W 帧时：

- 对 MKS 发送紧急停止。
- 对 BLD 发送停止或制动停止，并把速度设为 0。
- 清空固件中的已应用行走速度。
- STATUS 设置 `to=1`，故障码优先显示 `2001`。
- 不能继续执行上一条运动命令。

### 4.5 steer-before-drive

对每个使能轮组，执行顺序为：

1. 把车轮目标角换算为 MKS 绝对多圈坐标。
2. 向 MKS 写入目标坐标和速度上限。
3. 读取 MKS 当前累计坐标。
4. 只有角度误差进入配置容差，才向 BLD 写入非零转速。
5. 未到位时只停止 BLD，不能用“停止整个轮组”的方式把正在转向的 MKS 同时停掉。

## 5. 单位换算

### 5.1 转向

固件使用：

```text
目标坐标 = 零位坐标 + 方向符号 × angle_rad / (2π) × 30 × 16384
```

- `30` 是 NMRVS30 的转向减速比，只在 STM32 中使用一次。
- `16384` 是 MKS 手册中的单圈坐标计数。
- 每轮零位和方向符号在 `mwrs_config.h` 中独立配置。

### 5.2 行走

固件使用：

```text
motor_rpm = |velocity_mps| × 60 / (2π × wheel_radius) × drive_gear_ratio
```

当前占位参数为：

- 轮半径 `0.09 m`。
- 行走减速比 `20:1`。

这两个值尚未由实物确认。以当前值计算，`0.08 m/s` 约为 `170 rpm`。

## 6. STM32 负责人必须确认和填写的内容

### 6.1 `mwrs_config.h`

默认值如下：

```c
#define MWRS_ACTUATION_ENABLED               0
#define MWRS_HARDWARE_ESTOP_CONFIGURED       0
#define MWRS_DRIVER_PROTOCOLS_CONFIRMED       0
#define MWRS_WHEEL_CALIBRATION_CONFIRMED      0
```

在全部确认之前不得把 `MWRS_ACTUATION_ENABLED` 改成 1。启用真实输出前必须完成：

1. 实现并验证 `MWRS_HardwareEstopActive()`，然后把 `MWRS_HARDWARE_ESTOP_CONFIGURED` 改为 1。弱默认实现固定返回“急停激活”，遗漏板级覆盖时不会放行运动。
2. 核对 MKS 实物固件与 V1.0.9 Modbus RTU 寄存器兼容。
3. 把 MKS 设置为 Modbus RTU、115200、8N1，并配置 ID 1-4。
4. 把 BLD-305S 设置为内部通信模式、115200、8N1，并配置 ID 1-4。
5. 实测轮半径和行走减速比。
6. 对每轮记录 MKS 机械零位坐标和转向方向符号。
7. 对每轮记录 BLD 正方向符号。
8. 安装并验证 homing 开关，确认 `0x003B` 的非单圈 homing 状态可读。
9. 工程默认 `MWRS_REQUIRED_WHEEL_COUNT=4`，表示完整四轮能力；只有在 front_left 单轮台架上才临时改为 1。
10. 上述内容通过后，把驱动协议和轮组标定确认项改为 1，最后才允许把真实输出改为 1。

### 6.2 板级 USB 路径

2026-07-02 的 Pi 实测记录已经确认：

- Pi 枚举到 `STMicroelectronics STLINK-V3`。
- 设备为 `/dev/ttyACM0`，并有稳定的 `/dev/serial/by-id/...` 路径。
- 当前 LPUART1 固件能通过该 VCP 收到旧协议命令并返回 ACK/STATUS。

因此本次保留 `LPUART1 PC0/PC1 -> ST-LINK V3 VCP`，没有改成 MCU 原生 USB CDC。STM32 负责人仍应根据实际控制板原理图复核 PC0/PC1 与 ST-LINK VCP 的连接。

## 7. 已完成的验证

### 7.1 已通过

- 固定 W 帧 `*58782C06` 解析通过。
- 固定 ACK `*88B875B3` 编码通过。
- 固定 STATUS `*F18BB4D1` 编码通过。
- ACK/STATUS 已由当前 ROS 2 `serial_codec.py` 反向解析通过。
- CRC 错误、额外字段和越界速度拒绝测试通过。
- `0.1 rad` 转向坐标、30:1 减速比换算测试通过。
- `0.08 m/s -> 170 rpm` 测试通过。
- 正反方向和加速度斜坡测试通过。
- 新增 HAL 相关 C 文件已使用现有 STM32G4 HAL/CMSIS 头文件完成语法检查。

测试命令：

```powershell
powershell -NoProfile -ExecutionPolicy Bypass -File STM32/tests/run_host_tests.ps1
```

### 7.2 尚未验证

- 当前电脑没有安装 GNU Arm Embedded / STM32CubeIDE 命令行工具，因此本次未生成新的 `.elf`、`.bin` 或 `.hex`。
- 未在 STM32G474RE 实板上烧录本次固件。
- 未用紧凑协议做 Pi 到 STM32 的新一轮抓包。
- 未验证 MKS 实物固件版本和 `0x00F5` 绝对坐标命令。
- 未验证 BLD-305S ID、通信模式、极对数和真实故障响应。
- 未验证原点开关、方向、机械零位、负载和硬件急停。

因此，当前结论是“协议和控制代码可交付审查”，不是“整车硬件已经验收”。

## 8. 建议的审查与烧录顺序

1. STM32 负责人先阅读本文件和两份接口合同。
2. 在 CubeIDE 中打开 `STM32/STM32.ioc`，确认新增 `.c` 文件均进入构建。
3. 保持 `MWRS_ACTUATION_ENABLED=0` 完成第一次构建和烧录。
4. 只连接 Pi 与 STM32，用固定 W 向量验证 CRC、ACK、STATUS 和 0.5 秒 watchdog。
5. 只接 front_left 的 MKS ID 1 与 BLD ID 1，读取在线、homing 和故障状态。
6. 完成急停输入、零位、方向、轮半径和减速比配置。
7. 先把三个确认项设为 1，重新构建检查；最后才把 `MWRS_ACTUATION_ENABLED` 设为 1。
8. 悬空完成 front_left 小角度转向和低速正反转。
9. 验证 USB 拔出、W 帧停止、驱动器断线和故障时均能停止。
10. 单轮通过后恢复 `MWRS_REQUIRED_WHEEL_COUNT=4`，再逐轮接入整车。

## 9. 需要回交给 ROS 2 负责人的证据

- CubeIDE 无错误构建日志和固件 SHA256。
- 一组真实 W/A/S 抓包。
- 20 Hz W、逐帧 ACK、约 5 Hz STATUS 的持续测试结果。
- 0.5 秒 watchdog 实测时间。
- 每轮 MKS 固件版本、ID、零位、方向、homing 和堵转结果。
- 每轮 BLD ID、方向、实际 rpm 和 fault 结果。
- USB 断线、物理急停和驱动器断线时的停止证据。
